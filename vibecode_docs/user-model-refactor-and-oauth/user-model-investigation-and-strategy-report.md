  1. Current State

  PostgreSQL schema — public.user (migration 002)
  ┌────────┬────────┬───────────────────────────────────────────┐
  │ Column │  Type  │                Constraints                │
  ├────────┼────────┼───────────────────────────────────────────┤
  │ id     │ BIGINT │ PK, GENERATED BY DEFAULT AS IDENTITY      │
  ├────────┼────────┼───────────────────────────────────────────┤
  │ vkid   │ TEXT   │ NOT NULL, UNIQUE, CHECK(vkid <> '')       │
  ├────────┼────────┼───────────────────────────────────────────┤
  │ name   │ TEXT   │ NOT NULL, CHECK(name <> ''), LENGTH < 100 │
  ├────────┼────────┼───────────────────────────────────────────┤
  │ avatar │ TEXT   │ NOT NULL, CHECK(avatar != '')             │
  └────────┴────────┴───────────────────────────────────────────┘
  No created_at, updated_at, email, status, or role columns. No indexes beyond PK and the UNIQUE on vkid.

  Foreign keys referencing user(id):
  - encounter_store.user_id BIGINT (migration 003)
  - maps.user_id BIGINT (migration 006)

  MongoDB collections also store userID (as string) in: Character, Creature, MapTileCategory.

  Go model — models.User

  type User struct {
      ID     int    `json:"id"`
      VKID   string `json:"vkID"`
      Name   string `json:"name"`
      Avatar string `json:"avatar"`
  }

  Used everywhere: Redis session, context injection, API responses, ownership checks.

  Session architecture

  - Storage: Redis, key = UUID v4 session ID, value = JSON of FullSessionData
  - TTL: 30 days (hardcoded in handler)
  - FullSessionData contains both VK tokens (access_token, refresh_token, id_token) and User struct
  - Cookie: session_id, HttpOnly=true, no Secure flag, no SameSite

  VK OAuth flow

  1. Frontend sends authorization code → POST /api/auth/login
  2. Backend exchanges code → VK API → gets VKTokensData (access/refresh/id tokens)
  3. Backend fetches profile → VK API public_info → gets UserPublicInfo (user_id, first_name, last_name, email, phone, avatar)
  4. Upsert into public.user by vkid (create or update name/avatar if changed)
  5. Store full session (tokens + user) in Redis
  6. Return cookie + AuthResponse{isAuth, user}

  Key observation: email and phone are received from VK but discarded — they're in UserPublicInfo but never saved anywhere.

  2. Gaps & Opportunities

  Missing data
  ┌─────────────────────────┬───────────────────────────────────────────────────────────────────┐
  │           Gap           │                              Impact                               │
  ├─────────────────────────┼───────────────────────────────────────────────────────────────────┤
  │ No created_at           │ Can't tell when a user registered; no retention/cohort analytics  │
  ├─────────────────────────┼───────────────────────────────────────────────────────────────────┤
  │ No updated_at           │ Can't track profile sync frequency or debug stale data            │
  ├─────────────────────────┼───────────────────────────────────────────────────────────────────┤
  │ No last_login_at        │ Can't identify inactive users or detect suspicious activity       │
  ├─────────────────────────┼───────────────────────────────────────────────────────────────────┤
  │ No email                │ Available from VK but discarded; needed for Google/Yandex linking │
  ├─────────────────────────┼───────────────────────────────────────────────────────────────────┤
  │ No status field         │ Can't ban/soft-delete users without dropping the row              │
  ├─────────────────────────┼───────────────────────────────────────────────────────────────────┤
  │ No role                 │ No way to grant admin access if needed                            │
  ├─────────────────────────┼───────────────────────────────────────────────────────────────────┤
  │ VK hardcoded everywhere │ vkid column, VKApi interface, VKID field in User struct           │
  └─────────────────────────┴───────────────────────────────────────────────────────────────────┘
  Multi-provider blockers

  If Google/Yandex are added without refactor:

  1. Schema lock-in: vkid TEXT NOT NULL UNIQUE means every user must have a VK ID. A Google-only user can't exist.
  2. Usecase coupling: Login() calls vkApi.ExchangeCode() / vkApi.GetPublicInfo() directly — no abstraction for "any provider".
  3. Session stores VK tokens: FullSessionData.Tokens is VK-specific (access_token, refresh_token, id_token). These field names happen to overlap with
  Google/Yandex OAuth, but semantically they're VK tokens.
  4. User identity = vkid: Every CheckUser/CreateUser/UpdateUser query uses vkid as the lookup key. Adding google_id/yandex_id would require separate queries  
  or a redesign.
  5. Name is provider-dependent: fmt.Sprintf("%s %s", FirstName, LastName) is VK-specific formatting.

  Security notes

  - VK refresh_token stored as plaintext in Redis (inside FullSessionData JSON). Not encrypted.
  - Cookie lacks Secure and SameSite flags.
  - No token rotation mechanism.
  - No audit trail for logins.

  3. Proposed Model & Rollout Plan

  Recommended approach: users + user_identities table

  Why not just add columns (vk_id/google_id/yandex_id)?
  ┌─────────────────────────────────┬───────────────────────────────────┬──────────────────────────────────┐
  │            Criteria             │         Columns on users          │     Separate user_identities     │
  ├─────────────────────────────────┼───────────────────────────────────┼──────────────────────────────────┤
  │ Schema changes per new provider │ ALTER TABLE ADD COLUMN            │ None — just a new row            │
  ├─────────────────────────────────┼───────────────────────────────────┼──────────────────────────────────┤
  │ NULL columns                    │ Yes (google_id NULL for VK users) │ No — only rows that exist        │
  ├─────────────────────────────────┼───────────────────────────────────┼──────────────────────────────────┤
  │ Account linking                 │ Awkward                           │ Natural — multiple rows per user │
  ├─────────────────────────────────┼───────────────────────────────────┼──────────────────────────────────┤
  │ Query complexity                │ Simple                            │ One JOIN                         │
  ├─────────────────────────────────┼───────────────────────────────────┼──────────────────────────────────┤
  │ Migration risk                  │ Low                               │ Low (additive)                   │
  └─────────────────────────────────┴───────────────────────────────────┴──────────────────────────────────┘
  Recommendation: user_identities table. The project already has 3 planned providers and may add more. Columns approach leads to a sparse table and ALTER for  
  every new provider.

  10 fields total. Removed: vkid (moves to user_identities). Added: 6 new fields. Renamed: name → display_name, avatar → avatar_url.

  Proposed user_identities table (new)
  ┌──────────────────┬───────────────────────────┬───────────────────────────────────┐
  │      Column      │           Type            │            Constraints            │
  ├──────────────────┼───────────────────────────┼───────────────────────────────────┤
  │ id               │ BIGINT PK (identity)      │                                   │
  ├──────────────────┼───────────────────────────┼───────────────────────────────────┤
  │ user_id          │ BIGINT NOT NULL           │ FK → users(id) ON DELETE CASCADE  │
  ├──────────────────┼───────────────────────────┼───────────────────────────────────┤
  │ provider         │ TEXT NOT NULL             │ 'vk', 'google', 'yandex'          │
  ├──────────────────┼───────────────────────────┼───────────────────────────────────┤
  │ provider_user_id │ TEXT NOT NULL             │ External ID from provider         │
  ├──────────────────┼───────────────────────────┼───────────────────────────────────┤
  │ email            │ TEXT                      │ Email from this specific provider │
  ├──────────────────┼───────────────────────────┼───────────────────────────────────┤
  │ created_at       │ TIMESTAMPTZ DEFAULT now() │ When this identity was linked     │
  ├──────────────────┼───────────────────────────┼───────────────────────────────────┤
  │ last_used_at     │ TIMESTAMPTZ               │ Last login via this provider      │
  └──────────────────┴───────────────────────────┴───────────────────────────────────┘
  Indexes:
  - UNIQUE (provider, provider_user_id) — one account per provider
  - UNIQUE (user_id, provider) — one identity per provider per user
  - INDEX (user_id) — for joins

  7 fields. No token storage in DB — tokens stay in Redis session (short-lived). If refresh token persistence is needed later, add an encrypted
  encrypted_refresh_token column with application-level AES-256-GCM encryption and key rotation via config.

  Account linking rules

  1. Login with provider X → lookup user_identities by (provider, provider_user_id)
  2. If found → return user_id, update last_used_at
  3. If not found → check if same email (verified) exists on another identity → link to that user
  4. If no match → create new users row + user_identities row
  5. A user can have 1 identity per provider (enforced by unique constraint)

  Go model changes (conceptual)

  // models/user.go
  type User struct {
      ID            int        `json:"id"`
      DisplayName   string     `json:"displayName"`
      AvatarURL     string     `json:"avatarUrl,omitempty"`
      Email         string     `json:"email,omitempty"`
      EmailVerified bool       `json:"-"`           // internal
      Role          string     `json:"-"`           // internal
      Status        string     `json:"-"`           // internal
      CreatedAt     time.Time  `json:"-"`           // internal
      LastLoginAt   time.Time  `json:"-"`           // internal
  }

  // models/identity.go
  type UserIdentity struct {
      ID             int       `json:"-"`
      UserID         int       `json:"-"`
      Provider       string    `json:"provider"`
      ProviderUserID string    `json:"-"`           // never expose
      CreatedAt      time.Time `json:"-"`
  }

  Sensitive fields (Role, Status, ProviderUserID) are json:"-" — never leak to frontend.

  GDPR/Privacy (minimum viable)

  1. Don't store tokens in DB. Current approach (Redis session with TTL) is fine. Tokens auto-expire.
  2. Email as nullable. Users who don't consent to sharing email get NULL.
  3. Soft-delete. status = 'deleted' keeps FK integrity while hiding user data. For full GDPR deletion: nullify display_name, avatar_url, email and set status 
  = 'deleted'.
  4. No phone storage. VK provides phone — continue ignoring it.
  5. Audit trail. last_login_at + user_identities.last_used_at give minimal audit. Full audit log is out of scope for now.

  Rollout Plan (PRs in order)

  PR 1: Add timestamps to users

  Migration:
  ALTER TABLE public.user
      ADD COLUMN created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
      ADD COLUMN updated_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
      ADD COLUMN last_login_at TIMESTAMPTZ;

  Existing rows get created_at = now() (approximate — no historical data available). Acceptable.

  Code changes:
  - models.User — add CreatedAt, UpdatedAt, LastLoginAt fields
  - Repository — update CreateUser/UpdateUser queries to SET updated_at, last_login_at
  - Usecase — set last_login_at = now() on login
  - Delivery — no API changes (json:"-" on new fields)

  Risks: Minimal. Additive migration, no breaking changes. Existing queries continue to work.
  Testing: Unit tests for updated repo methods. Integration test for migration on clean DB.

  PR 2: Add status, role, rename columns

  Migration:
  ALTER TABLE public.user
      ADD COLUMN status TEXT NOT NULL DEFAULT 'active',
      ADD COLUMN role   TEXT NOT NULL DEFAULT 'user';
  ALTER TABLE public.user RENAME COLUMN name   TO display_name;
  ALTER TABLE public.user RENAME COLUMN avatar TO avatar_url;
  ALTER TABLE public.user ALTER COLUMN avatar_url DROP NOT NULL;

  Code changes:
  - models.User — rename Name → DisplayName, Avatar → AvatarURL, add Status, Role
  - All queries referencing name/avatar columns — update to display_name/avatar_url
  - Auth middleware — add check: if status != 'active', reject with 403
  - Delivery responses — update JSON field names (breaking API change — coordinate with frontend)

  Risks: Column rename is a breaking change for running queries. Mitigation: deploy in maintenance window or use a two-step migration (add new columns →       
  backfill → drop old).
  Testing: Unit tests for status check in middleware. Handler tests for JSON field name changes.

  PR 3: Create user_identities table, migrate VK data

  Migration (up):
  CREATE TABLE public.user_identity (
      id                BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
      user_id           BIGINT NOT NULL REFERENCES public.user(id) ON DELETE CASCADE,
      provider          TEXT NOT NULL,
      provider_user_id  TEXT NOT NULL,
      email             TEXT,
      created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
      last_used_at      TIMESTAMPTZ,
      UNIQUE (provider, provider_user_id),
      UNIQUE (user_id, provider)
  );

  -- Migrate existing VK identities
  INSERT INTO public.user_identity (user_id, provider, provider_user_id, created_at)
  SELECT id, 'vk', vkid, created_at FROM public.user;

  Then (separate migration or same):
  ALTER TABLE public.user ADD COLUMN email TEXT;
  ALTER TABLE public.user ADD COLUMN email_verified BOOLEAN NOT NULL DEFAULT FALSE;
  ALTER TABLE public.user DROP COLUMN vkid;

  Code changes:
  - New IdentityRepository interface: FindByProvider(ctx, provider, providerUserID), CreateIdentity(ctx, identity), UpdateLastUsed(ctx, identityID)
  - AuthRepository.CheckUser changes from WHERE vkid = $1 to a JOIN or to using IdentityRepository
  - models.User drops VKID field
  - Usecase login flow refactored: lookup identity → get/create user → create/update identity

  Risks: This is the biggest migration. DROP COLUMN vkid removes the old lookup path. Mitigation:
  1. Keep vkid column temporarily (nullable) during transition
  2. Deploy new code that reads from user_identity
  3. Next migration drops vkid

  Testing: Integration test: create user via old schema, run migration, verify identity row exists and login works. Unit tests for new IdentityRepository.     

  PR 4: Abstract OAuth provider interface

  Code changes:
  - New interface OAuthProvider:
  type OAuthProvider interface {
      ExchangeCode(ctx context.Context, req *OAuthExchangeRequest) (*OAuthTokens, error)
      GetUserInfo(ctx context.Context, tokens *OAuthTokens) (*OAuthUserInfo, error)
      ProviderName() string
  }
  - OAuthTokens / OAuthUserInfo — provider-agnostic structs
  - Existing VKApi becomes an implementation of OAuthProvider
  - Login usecase takes provider parameter, selects the right OAuthProvider
  - POST /api/auth/login adds provider field to request (default "vk" for backwards compat)

  Risks: Refactor of auth interfaces. All existing tests need updating.
  Testing: All auth unit tests rewritten against OAuthProvider interface. VK flow tested with mock OAuthProvider.

  PR 5: Add Google OAuth

  Code changes:
  - Implement GoogleOAuthProvider (exchange via https://oauth2.googleapis.com/token, user info via https://www.googleapis.com/oauth2/v3/userinfo)
  - Register in DI/wiring
  - Config: add GOOGLE_CLIENT_ID, GOOGLE_SECRET env vars
  - Frontend route: POST /api/auth/login with {"provider": "google", "code": "...", ...}

  No migration needed — user_identity table already supports any provider.

  Testing: Unit tests with mock OAuthProvider. Integration test with real Google sandbox (optional).

  PR 6: Add Yandex OAuth

  Same pattern as PR 5. Yandex endpoint: https://oauth.yandex.ru/token, user info: https://login.yandex.ru/info.

  PR 7: API/DTO cleanup

  - Update AuthResponse to include provider info
  - Add GET /api/auth/identities — list linked providers for current user
  - Add POST /api/auth/link — link additional provider to existing account
  - Add DELETE /api/auth/unlink/{provider} — unlink a provider (with guard: can't unlink last one)

  This plan is additive and non-destructive at each step. PRs 1-2 can ship independently and provide value (timestamps, status) before the multi-provider work
